# План: десктоп-приложение «GeoJSON Zones» (Windows, .exe)

## Контекст

Сейчас проект — это одностраничный HTML-инструмент (`geojson to xlxs2.html`), который
принимает GeoJSON и отдаёт Excel с двумя столбцами координат. Вся ценная логика —
рекурсивный разбор любых типов геометрии GeoJSON и извлечение точек — в функции
`extractLonLat` (строки 247–380) и хелперах `makePointPusher`, `processPolygon`,
`processLineString`.

GeoJSON-файлы — это **зоны** (полигоны), очерченные в агломерации города. Они нужны
сервису расчёта стоимости перевозки и проверки входимости адреса клиента в зону.

Нужна полноценная Windows-программа (устанавливается через .exe), которая:
1. Хранит зоны и группирует их по городам (справочник городов = «папки»).
2. Даёт CRUD по городам и зонам, перенос зон между городами.
3. Показывает иерархический журнал «город → зоны».
4. По каждой зоне — кнопки «скачать GeoJSON» и «скачать XLSX».
5. Отображает полигон зоны на карте.
6. Добавляет зону в город из файла (диалог открытия .geojson).
7. Ведёт лог действий в отдельной вкладке.

**Ключевое изменение модели данных:** текущий инструмент выбрасывает всё кроме
координат. Чтобы поддержать «скачать GeoJSON», нужно хранить **исходный GeoJSON
целиком**, а XLSX и точки генерировать из него по запросу.

## Решения (подтверждены пользователем)

- **Стек:** Electron + electron-builder (NSIS-установщик .exe). Переиспользуем
  существующий JS-код парсинга.
- **Хранилище:** локальная БД SQLite (источник истины). Файлы GeoJSON/XLSX
  генерируются по требованию.
- **Карта:** Leaflet + тайлы OpenStreetMap (требуется интернет при просмотре карты).
- **Интерфейс:** современный, чистый UI (см. раздел «Дизайн интерфейса»).
- **Переименование без изменения файлов:** имя в программе и содержимое файла —
  разные сущности (см. раздел «Переименование»).
- **Git/GitHub:** все изменения коммитятся и пушатся в `origin`
  (`git@github.com:Nikitgolubev/geojson-to-xlsx.git`) после каждого этапа.
- **Фиксация плана:** этот план копируется в репозиторий как `PLAN.md` и ведётся вместе с кодом.
- **Задел под macOS:** сейчас собираем только Windows, но код и конфиг пишем
  кроссплатформенно, чтобы позже добавить сборку под macOS без переделок
  (см. раздел «Задел под macOS»).

```
│   ├── main.js               # главный процесс: окно, IPC, доступ к БД и ФС
│   ├── preload.js            # безопасный мост contextBridge → window.api

│   ├── index.html            # каркас вкладок
│   ├── styles.css            # стили (взять за основу текущие из geojson to xlxs2.html)
│   ├── app.js                # роутинг вкладок, инициализация
│   ├── geojson.js            # ПЕРЕНОС extractLonLat и хелперов «как есть»
│   ├── xlsx-export.js        # точки → XLSX (логика из processSingleFile)
│   ├── views/
│   │   ├── catalog.js        # справочник городов (CRUD)
│   │   ├── zones.js          # иерархический журнал город→зоны + кнопки
│   │   ├── map.js            # Leaflet-карта, рендер полигона зоны
│   │   └── log.js            # вкладка журнала действий
│   └── vendor/               # локальные копии xlsx, leaflet (без CDN — работа офлайн UI)
└── build/                    # иконки и ресурсы установщика
```

**Принцип разделения, чтобы не было постоянного рефакторинга:**
- `electron/db.js` — единственное место, знающее про SQL. UI ходит только через `window.api`.
- `src/geojson.js` и `src/xlsx-export.js` — чистые функции без DOM, переиспользуемы и тестируемы.
- Каждая вкладка (`views/*.js`) самодостаточна и общается с данными только через `window.api`.
- Добавление новой функции = новый метод в `db.js` + новый вызов `api` + правка одной вкладки. Остальное не трогается.

## Переименование без изменения файлов (важное требование)

Имя зоны/города в программе — это **только метка в БД** (поле `name`), отдельная от
содержимого файла:
- Переименование зоны меняет **только** `zones.name`. Поля `geojson` и
  `source_filename` (исходное имя файла) **не трогаются**. Внутренние свойства
  GeoJSON (`properties`, координаты и т.д.) остаются байт-в-байт.
- Переименование города меняет **только** `cities.name`. Привязанные зоны и их
  файлы не меняются.
- При экспорте по умолчанию имя файла = текущее имя в программе (удобно), но рядом
  даём опцию «сохранить исходное имя файла» (`source_filename`). Содержимое экспорта
  всегда = исходный `geojson`, без модификаций.

Итог: пользователь свободно переименовывает города и зоны в программе, и это
никогда не портит и не переписывает сами GeoJSON/XLSX данные.

- Левая боковая панель навигации по вкладкам (Города · Зоны · Карта · Журнал)
  вместо классических табов сверху — выглядит современнее.
- Светлая тема по умолчанию, опционально тёмная (CSS-переменные для цветов).
- Скруглённые карточки, мягкие тени, системный шрифт (как в текущем CSS),
  единый accent-цвет.
- Иконки (через встроенный SVG-набор, напр. Lucide — копируем нужные локально, без CDN).
- Иерархия город→зоны в виде раскрывающихся карточек/дерева с hover-состояниями.
- Подтверждения удаления — модальное окно, а не `alert`/`confirm` (современнее и единообразно).
- Тосты/уведомления об успехе действий вместо примитивного лога на экране.
- Адаптивная сетка: карта и список рядом на широком окне, стопкой на узком.
- **Drag&drop-зона** для загрузки .geojson с визуальным откликом при наведении файла.
- **Подсветка зон без города** (warning-фон + бейдж «не выбран город»).
- Бейджи дат у зоны (создана / GeoJSON изменён / XLSX сгенерирован).

Базу стилей берём из `geojson to xlxs2.html:8-125`, расширяем CSS-переменными и
компонентами. Без тяжёлых UI-фреймворков — чистый CSS, чтобы установщик оставался лёгким.

## Модель данных (SQLite)

```sql
CREATE TABLE cities (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL UNIQUE,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE zones (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  city_id INTEGER REFERENCES cities(id) ON DELETE SET NULL,  -- NULL = «не выбран город»
  name TEXT NOT NULL,
  geojson TEXT NOT NULL,             -- исходный GeoJSON целиком (источник истины)
  point_count INTEGER,               -- кэш числа точек, считается при импорте
  source_filename TEXT,              -- имя исходного файла, если импортирован
  created_at TEXT NOT NULL DEFAULT (datetime('now')),        -- дата создания зоны
  geojson_updated_at TEXT NOT NULL DEFAULT (datetime('now')),-- дата посл. изменения GeoJSON
  xlsx_generated_at TEXT                                     -- дата посл. генерации/скачивания XLSX (NULL пока не делали)
);

CREATE TABLE action_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts TEXT NOT NULL DEFAULT (datetime('now')),
  level TEXT NOT NULL,          -- info / warn / error
  message TEXT NOT NULL
);
```

БД лежит в `app.getPath('userData')/zones.db` — переживает обновления приложения.

**Даты (показываем в интерфейсе):**
- `cities.created_at` — дата создания города.
- `zones.created_at` — дата создания зоны.
- `zones.geojson_updated_at` — обновляется при любом изменении `geojson` (импорт/замена).
- `zones.xlsx_generated_at` — обновляется при каждой генерации/скачивании XLSX; пока не делали — `NULL` (показываем «—»).
- Переименование зоны/города эти даты **не** меняет (имя ≠ содержимое).

**Состояние «не выбран город»:**
- `zones.city_id = NULL` означает, что зона добавлена, но город ещё не назначен.
- Возникает при массовой/drag&drop загрузке и при удалении города (зоны открепляются).
- В UI такие зоны выводятся **отдельной секцией сверху**, подсвечены цветом
  (напр. янтарный/warning фон) и помечены бейджем **«не выбран город»**.

## IPC API (preload → `window.api`)

- `cities.list() / create(name) / rename(id,name) / delete(id)` — delete открепляет зоны (city_id → NULL)
- `zones.listByCity(cityId) / listUnassigned() / get(id)`
- `zones.create(name, geojson, cityId?)` — `cityId` необязателен; без него зона создаётся «без города»
- `zones.rename(id,name) / assignCity(id, cityId) / move(id, newCityId) / delete(id)`
  (`assignCity` назначает город откреплённой зоне; `cityId=null` — снова открепить)
- `zones.importFiles(fileList, cityId?)` — массовая загрузка (диалог или drag&drop);
  валидирует каждый, сохраняет; без `cityId` все попадают в «не выбран город»
- `zones.exportGeojson(id) / zones.exportXlsx(id)` → диалог «Сохранить как» (exportXlsx обновляет `xlsx_generated_at`)
- `log.list(limit) / log.append(level,message) / log.clear()`

## Пошаговая реализация

> **Git после каждого этапа:** по завершении этапа — `git add -A`,
> осмысленный коммит, `git push origin main`. Так каждое изменение уходит в GitHub
> и история шагов сохраняется. (`gh` не установлен — пушим обычным `git push` по SSH.)

### Этап -1. Чистка репозитория и документация (ДО программирования)
1. Удалить из репозитория лишние эксперименты с кнопками (к проекту не относятся):
   `index.html`, `index.html.txt`, `index2.html` (`git rm`).
2. Сохранить как референс логики: `geojson to xlxs2.html` (основной) и
   `geojson to xlxs.html` — пока не удаляем, из них переносим `extractLonLat`.
   После переноса логики (Этап 2) переместим их в папку `legacy/` или удалим отдельным коммитом.
3. Создать **первую версию документации** (это и есть «первая версия проекта в git»):
   - `README.md` — назначение, что делает программа, как собрать/запустить, статус (в разработке).
   - `PLAN.md` — копия этого плана.
   - `CLAUDE.md` — файл контекста (стек, структура, модель данных, `window.api`, решения, задел под macOS).
   - `.gitignore` — `node_modules/`, `dist/`, `*.db`.
4. Коммит и `git push origin main` — фиксируем чистый старт **до** написания кода приложения.

### Этап 0. Каркас проекта
1. `npm init`, установить: `electron`, `electron-builder`, `better-sqlite3`, `xlsx`, `leaflet`.
2. Настроить `main.js` (создание окна, `contextIsolation: true`, загрузка `src/index.html`).
3. `preload.js` с `contextBridge.exposeInMainWorld('api', …)`.
4. Скрипты в `package.json`: `start` (electron .), `dist:win` (electron-builder --win).
5. Проверка: `npm start` открывает пустое окно.

### Этап 1. Слой данных
1. `db.js`: открыть/создать БД, выполнить схему (миграция «создать если нет»).
2. Реализовать все методы репозитория (cities/zones/log).
3. Подключить IPC-обработчики в `main.js`, пробросить в `preload.js`.
4. Проверка: из DevTools-консоли renderer вызвать `await window.api.cities.create('Тест')` и `list()`.

### Этап 2. Перенос логики GeoJSON
1. Скопировать `extractLonLat` + хелперы в `src/geojson.js` как ES-модуль (экспорт функций). Логику не менять — она проверена.
2. `src/xlsx-export.js`: функция `pointsToWorkbook(points)` и `geojsonToXlsxBytes(geojson, opts)` на базе текущего кода `processSingleFile`/`processMultipleFiles`.
3. Проверка: юнит-вызов на тестовом полигоне даёт ожидаемое число точек.

### Этап 3. Вкладка «Справочник городов» (catalog.js)
1. Список городов, форма «добавить», inline-редактирование имени.
2. Показывать **дату создания города** (`created_at`) и число зон.
3. **Удаление города** — модальное подтверждение; зоны НЕ удаляются, а открепляются
   (становятся «не выбран город») — предупредить об этом в модалке.
4. **Переименование города** — меняет только `cities.name`, файлы зон не затрагиваются.
5. Каждое действие пишет в `action_log`.
6. Проверка: добавить/переименовать/удалить город (зоны открепились, не удалились), перезапустить — данные на месте.

### Этап 4. Вкладка «Зоны» — иерархический журнал + drag&drop (zones.js)

Рабочий поток: **сначала добавляют зону(-ы), потом назначают город.**

1. **Зона drag&drop:** заметная область «Перетащите сюда .geojson файлы» вверху вкладки
   (+ кнопка «Выбрать файлы» как альтернатива). Принимает один или несколько файлов.
   - Каждый файл валидируется (`JSON.parse` + `extractLonLat`); при 0 точек — предупреждение, но сохранить можно.
   - Загруженные зоны создаются **без города** (`city_id = NULL`) → `zones.importFiles`.
2. **Секция «Не выбран город»** — отдельным блоком сверху, подсвечена цветом (warning-фон),
   у каждой зоны бейдж **«не выбран город»** и селект **«Назначить город»** → `zones.assignCity`.
3. **Иерархия город → зоны** ниже: раскрывающиеся узлы городов с их зонами.
4. По каждой зоне:
   - Кнопки **Скачать GeoJSON**, **Скачать XLSX**, **На карте**, **Переименовать**, **Удалить**.
   - **Переместить/назначить город** — селект городов (вкл. «без города») → `zones.assignCity`/`move`.
   - Показ дат: **создана** (`created_at`), **GeoJSON изм.** (`geojson_updated_at`),
     **XLSX сген.** (`xlsx_generated_at` или «—»).
5. **Переименование зоны** — меняет только `zones.name`; `geojson`, `source_filename` и даты содержимого не трогаются.
6. **Удаление зоны** — модальное подтверждение → `zones.delete`.
7. Проверка:
   - drag&drop нескольких файлов → все попадают в «не выбран город», подсвечены, с бейджем;
   - назначить город одной из них → переходит в нужный узел, бейдж исчезает;
   - переименовать зону → скачать GeoJSON: содержимое идентично исходному;
   - даты отображаются и обновляются (скачали XLSX → появилась дата генерации).

### Этап 5. Вкладка «Карта» (map.js)
1. Leaflet-карта с тайлами OSM (локальная копия leaflet.js/css в `vendor`, тайлы — онлайн).
2. По выбору зоны: `L.geoJSON(geojson)` + `fitBounds`. Поддержать Polygon/MultiPolygon/LineString/Point.
3. Проверка: выбранная зона корректно рисуется и центрируется.

### Этап 6. Вкладка «Журнал действий» (log.js)
1. Таблица из `action_log` (время, уровень, сообщение), автообновление, кнопка «Очистить».
2. Проверка: действия из других вкладок появляются в журнале.

### Этап 7. Сборка установщика (Windows)
1. Конфиг `build` в `package.json`: `appId`, `productName`, `win.target: nsis`, иконка, `oneClick:false` (выбор папки), `perMachine` по желанию.
2. `npm run dist:win` → `dist/…Setup.exe`.
3. Важно: `better-sqlite3` — нативный модуль, требует пересборки под Electron (`electron-rebuild` или `nativeRebuild` в electron-builder). Заложить это.
4. Проверка: установить .exe на чистой Windows-машине/VM, пройти сценарии.

## Задел под macOS (сейчас не собираем, но не мешаем будущему)

Цель — чтобы добавить mac-версию позже можно было почти без переделок:
- **Пути и ФС:** только `app.getPath('userData')` и `path.join` — никаких
  захардкоженных `C:\...` или обратных слешей.

  легко дописать секцию `mac` (`dmg`/`zip`, `category`) и скрипт `dist:mac`.
  Сейчас добавляем только `dist:win`, но оставляем общий `dist` нейтральным.
- **Нативный модуль:** `better-sqlite3` собирается под каждую ОС отдельно —
  отметить в `CLAUDE.md`, что mac-сборку нужно делать/пересобирать на macOS.
- **UI и шрифты:** системный шрифт-стек (уже кроссплатформенный), без win-only API.
- **Меню/ярлыки:** не завязываться на специфику Windows в коде главного процесса.
- В `CLAUDE.md` отдельный пункт «Будущее: macOS» с этим чек-листом.

> macOS-сборку (`dmg`) в текущем объёме НЕ делаем — только не закрываем себе путь к ней.

## Проверка (end-to-end)

1. `npm start` — приложение открывается, 4 вкладки.
2. **Drag&drop** нескольких .geojson → зоны появляются в секции «не выбран город», подсвечены, с бейджем.
3. Назначить зоне город → она переходит в узел города, бейдж пропадает; даты создания/изменения отображаются.
4. «Скачать GeoJSON» возвращает исходный файл байт-в-байт; «Скачать XLSX» открывается в Excel, обновляется дата генерации XLSX.
5. «На карте» рисует полигон в правильном месте.
6. Переименовать зону и город в программе → скачать GeoJSON: содержимое не изменилось, даты содержимого не сдвинулись.
7. Удалить город → его зоны **открепляются** (не удаляются), уходят в «не выбран город».
8. Удалить зону — исчезает после подтверждения.
9. Вкладка журнала отражает все действия.
10. `npm run dist:win` собирает Setup.exe; установка на Windows-VM проходит, данные сохраняются между запусками.
11. После каждого этапа изменения запушены в `origin` (GitHub).
