## Веб-миграция — постоянный контекст

У проекта **две линии**, и агент обязан помнить, в какой работает:

- **`main` — десктопная Electron-версия.** Сохраняется и развивается как самостоятельный
  продукт. Веб-код сюда **не** попадает.
- **`web` — браузерная многопользовательская версия** (Windows + IIS, БД — MS SQL Server).
  Вся веб-разработка идёт в этой ветке.
- Линии могут расходиться; фичи одной не обязаны попадать в другую.

### Инварианты (нарушать нельзя)

1. **`main` всегда собирает рабочий `.exe`.** Веб-работа ведётся в `web` и не ломает `main`.
   Обратный мерж — только после прохождения десктоп-сборки и смоуков.
2. **Два шва — единственный способ делить код между линиями, без дублирования логики:**
   - *Шов данных:* доступ к БД только через интерфейс `ZonesRepository`. Реализации —
     `SqliteRepository` (десктоп) и `MssqlRepository` (веб).
   - *Шов транспорта:* `renderer` зовёт API-клиент, а не `ipcRenderer` напрямую. Реализации —
     IPC (десктоп) и `fetch` (веб).
3. **Бизнес-логика не зависит от Electron и от конкретной БД** — живёт в общем ядре
   (`shared/`): разбор GeoJSON, XLSX, ray-casting, геокодер, построение полигона.
4. **Никаких секретов в репозитории.** Пароль к БД, ключ сессий и пр. — только в переменных
   окружения; значения не коммитить, держать `.env.example` без значений.
5. **Рефакторинг ядра — единственное, что можно влить в `main`** (он не меняет поведение
   десктопа). Веб-сервер, IIS-конфиги, авторизация в `main` не вливаются.

### Дейт перед обратным мержем в `main`

```bash
npm run dist:win   # должен дать рабочий .exe (только на Windows)
npm run test:db && npm run test:func && npm run test:ui
```

Если десктоп-сборка или смоуки падают — мерж в `main` запрещён.
