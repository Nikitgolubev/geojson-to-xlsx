# Веб-миграция GeoJSON Zones — точка входа

> ⚠️ **План в резерве. Реализацию НЕ начинаем — ждём отдельной команды.**
> Этот пакет документов подготовлен «до востребования». Никакого кода, веток и
> тегов по нему пока не создаётся.

## Выжимка: что и зачем

**Приложение.** Десктоп на Electron (`better-sqlite3` + SheetJS + Leaflet). Хранит
зоны (полигоны GeoJSON), сгруппированные по городам; есть проверка вхождения адреса
в зону (ray-casting), рисование полигонов, экспорт GeoJSON/XLSX, синхронизация данных
с GitHub. Версия стабильная, есть установщик `.exe`.

**Цель миграции.** Добавить **браузерную многопользовательскую версию** (5–10
пользователей, высокой нагрузки нет). Десктоп при этом сохраняется и развивается
отдельно.

## Ключевые решения

1. **Не переписывать с нуля.** Electron — это уже браузер + Node, поэтому переход =
   «разрезать приложение на две половины и разнести по машинам»: frontend → в браузер,
   backend + БД → на сервер, обмен по HTTP вместо IPC.
2. **Два шва, чтобы десктоп и веб делили код без дублирования логики:**
   - *Шов данных* — доступ к БД только через интерфейс `ZonesRepository`; реализации
     `SqliteRepository` (десктоп) и `MssqlRepository` (веб).
   - *Шов транспорта* — `renderer` зовёт API-клиент, а не `ipcRenderer` напрямую;
     реализации IPC (десктоп) и `fetch` (веб).
3. **База веб-версии — уже развёрнутый MS SQL Server** (пакет `mssql`). Файл SQLite
   нельзя открыть в MS SQL напрямую — данные **переносятся скриптом**, а не подключаются.
4. **Стратегия веток.** `main` — десктоп (сохраняется, развивается сам, веб-код сюда не
   попадает). `web` — отдельная долгоживущая ветка для веба. Общее между линиями —
   только рефакторинг ядра.
5. **Размещение — Windows + IIS.** Node работает Windows-службой (nssm) на localhost;
   IIS впереди как reverse-proxy (URL Rewrite + ARR): отдаёт статику, проксирует `/api`,
   терминирует HTTPS.
6. **Инвариант.** `main` всегда собирает рабочий `.exe`; веб-работа в `web` его не ломает.

## Когда начнём — порядок шагов (будущие действия, не сейчас)

1. Прочитать `README.md` и текущий `CLAUDE.md` проекта.
2. Вставить раздел из `CLAUDE-section.md` в `CLAUDE.md` (убрать верхнюю строку-инструкцию).
3. Убедиться, что `PLAN.md` лежит в репозитории (это источник задач).
4. **T0** — зафиксировать baseline (последний стабильный десктоп — тег **`v0.9.2`**) и
   создать ветку `web` от `main`. Вся дальнейшая веб-работа — в `web`.
5. Выполнять план с **T1** по ID, по порядку; перед переходом к следующей задаче
   прогонять её команду-проверку. Не нарушать инварианты из `CLAUDE.md`.

## Где останавливаться и спрашивать человека

- Реквизиты MS SQL, домен, сертификаты, установка IIS-модулей, любые ops-действия (T5, релиз).

  централизованы в MS SQL.
- Любая правка, после которой `npm run dist:win` из `main` перестаёт давать рабочий `.exe`.
