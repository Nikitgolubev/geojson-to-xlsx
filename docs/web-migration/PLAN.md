# Веб-миграция GeoJSON Zones — рабочий план задач (T0–T7)

> ⚠️ **План в резерве.** Реализацию не начинаем без отдельной команды.
> Когда начнём: выполнять задачи по порядку их ID; после каждой прогонять указанную
> проверку и не переходить к следующей, пока текущая не проходит «Готово, когда».

## Перед началом

1. Прочитать `README.md` (точка входа) и корневой `CLAUDE.md` — стек, договорённости, архитектура.
2. Вся веб-работа — в отдельной ветке `web`; ветку `main` (десктоп) не ломать.
3. Стек: Electron + electron-builder, `better-sqlite3` (SQLite), SheetJS (xlsx), Leaflet.
   Папки: `shared/`, `src/`, `electron/`, `scripts/`, `test/`.
4. Целевая среда веба: **Windows + IIS**, БД — **MS SQL Server** (пакет `mssql`). Масштаб: 5–10 пользователей.

## Краткий обзор по фазам

**Переиспользуется почти без изменений:**
- Бизнес-логика ядра: разбор GeoJSON, XLSX-экспорт, ray-casting (вхождение точки),
  геокодер, построение полигона — уже в `shared/` (`shared/geojson.js` и др.).
- Весь UI-слой: Leaflet, карта, темы, поиск, вкладки, рисование полигона — это уже
  «веб-код» (HTML/CSS/JS в `src/`).
- Существующие чистые node-тесты логики.

**Действительно новое:**
- Шов данных: интерфейс `ZonesRepository` + реализация под MS SQL; перенос данных.
- Шов транспорта: API-клиент (IPC ↔ `fetch`).
- HTTP-сервер и эндпоинты.
- Авторизация и многопользовательность.
- Инфраструктура: Windows-служба + IIS (reverse-proxy, HTTPS), бэкапы MS SQL.

## T0 — Фиксация baseline и ветки

- **T0.1** Поставить git-тег на текущую стабильную версию как опорную точку десктопа.
  Базовый тег: **`v0.9.2`** (последний стабильный десктоп).
- **T0.2** Создать ветку `web` от стабильного `main`. Вся веб-работа — в ней.
- **T0.3** Зафиксировать в `CLAUDE.md`: стратегию веток (десктоп в `main` живёт сам,
  веб в `web`) и инвариант «`main` всегда собирает `.exe`» (вставить раздел из
  `CLAUDE-section.md`).
- **Готово, когда:** есть тег baseline, есть ветка `web`, CI собирает `.exe` из `main`.

## T1 — Выделение общего ядра и шва данных

> T1 — единственная фаза, которую желательно влить и в `main` (не меняет поведение десктопа).

- **T1.1** Перенести чистую логику в `shared/` (или `core/`): разбор GeoJSON, генерация
  XLSX, ray-casting, разбор геокодера, построение полигона. Без `require('electron')`,
  без `better-sqlite3`. (Значительная часть уже в `shared/geojson.js`.)
  - *Проверка:* `npm run test:geojson && npm run test:pip && npm run test:geocode && npm run test:polygon`.
- **T1.2** Описать интерфейс репозитория `ZonesRepository` (CRUD городов, CRUD зон,
  назначить/очистить город у зоны, выборки, счётчик зон без города).
- **T1.3** Реализовать `SqliteRepository` поверх текущего `better-sqlite3`; перенести в
  него весь существующий SQL (сейчас он в `electron/db.js`).
- **T1.4** Переключить main-процесс на работу через `ZonesRepository` вместо прямых
  `db.prepare(...)`.
  - *Проверка:* `npm run test:db && npm run test:func`.
- **Готово, когда:** десктоп ведёт себя как раньше, все тесты зелёные, прямых вызовов БД
  вне `SqliteRepository` не осталось.

## T2 — Веб-backend, БД и перенос данных (MS SQL)

- **T2.1** Node-сервер на Express в `server/`. Точка входа `server/index.js`, конфиг из `process.env`.
- **T2.2** SQL-схема в `server/schema.sql`: `cities`, `zones` (`city_id INT NULL` =
  «без города»), `id INT IDENTITY(1,1)`, `geojson NVARCHAR(MAX)`,
  `created_at/updated_at DATETIME2 DEFAULT SYSUTCDATETIME()`.
- **T2.3** Реализовать `MssqlRepository` — вторая реализация `ZonesRepository` поверх
  `mssql`, подключение через пул. Учесть T-SQL: `OUTPUT INSERTED.id` вместо
  `lastInsertRowid`, именованные параметры `@param`, `TOP`/`OFFSET FETCH` вместо `LIMIT`,
  `MERGE` для upsert.
- **T2.4** Эндпоинты `/api/...`, зеркалящие текущие IPC-каналы один-к-одному. Каждый
  роут зовёт ту же логику ядра + репозиторий.
- **T2.5** Загрузка `.geojson` (multipart): принять → распарсить ядром (T1.1) → записать
  через репозиторий. Файл на диске не хранить.
- **T2.6** `.env.example` (`DB_HOST`, `DB_USER`, `DB_PASS`, `DB_NAME`, `SESSION_SECRET`,
  `PORT`) без значений.
- **T2.7 — Подготовка БД и перенос данных в MS SQL.** Это и есть «открытие данных в SQL».
  - Важно: файл SQLite (`.db`) нельзя открыть в MS SQL напрямую — данные **переносятся**, а не подключаются.
  - **Создать БД и схему:** в SSMS подключиться → создать БД (напр. `geozones`) →
    выполнить `server/schema.sql`. Либо из консоли:
    `sqlcmd -S <server> -d geozones -i server/schema.sql`.
  - **Перенести данные.** Рекомендуемый способ — одноразовый Node-скрипт
    `scripts/migrate-to-mssql.js`, который читает через `SqliteRepository` (T1.3) и пишет
    через `MssqlRepository` (T2.3). Плюс: переиспользует обе реализации, проходит ту же
    валидацию, не нужно руками чинить диалект и экранирование. Сохранить связи `city_id`;
    GeoJSON писать как есть в `NVARCHAR(MAX)`.
  - *Альтернатива (если нужен снимок в репозитории):* сгенерировать `.sql` с `INSERT`-ами
    и выполнить в SSMS. Осторожно с экранированием кавычек внутри GeoJSON.
  - *Не рекомендуется:* CSV + `BULK INSERT`/`bcp` — GeoJSON содержит запятые и кавычки, разбор ломается.
  - **Проверка:** число городов и зон в источнике == в MS SQL; на нескольких зонах
    проверить `ISJSON(geojson) = 1`.
- **Готово, когда:** сервер стартует, подключается к MS SQL; схема создана; существующие
  данные перенесены и сверены; каждый эндпоинт отвечает (curl/Postman/`npm run test:api`).

- **T3.3** Drag&drop `.geojson`: в вебе — отправка на эндпоинт загрузки (T2.5).
- **T3.4** Адаптировать фичи на локальную ФС: «экспорт в папки» и «скачать в выбранную

- **T3.5** Проверить, что геокодер Nominatim и геолокация требуют HTTPS (учтётся в T5).

## T5 — Размещение на Windows + IIS

> Принцип: Node работает Windows-службой на локальном порту (напр. 3000). IIS впереди как
> reverse-proxy: отдаёт статику frontend, проксирует `/api` на Node, терминирует HTTPS.
> (Модуль `iisnode` фактически не поддерживается — используем reverse-proxy через ARR.)
> Поднятие серверов, домен и сертификаты делает человек; задача агента — подготовить
> конфиги и `DEPLOY-IIS.md`.

- **T5.1** Запустить Node как Windows-службу: `nssm install GeoZonesWeb` — путь к `node`,
  аргумент `server\index.js`, рабочая папка проекта, env из `.env`. Служба поднимается при
  загрузке Windows и перезапускается при падении. (Альтернатива: `pm2` + `pm2-windows-startup`.)
- **T5.2** Установить компоненты IIS: роль **Web Server (IIS)** + модули **URL Rewrite** и
  **Application Request Routing (ARR)**. В ARR включить прокси: *Server → Application Request
  Routing Cache → Server Proxy Settings → Enable proxy*.
- **T5.3** Создать сайт в IIS: физический путь — папка со статикой frontend (сборка из T3.2);
  биндинги на домен (80 и 443).
- **T5.4** Положить `web.config` в корень сайта с правилами URL Rewrite: `/api/*`
  проксировать на Node, остальное отдавать как статику (с фолбэком на `index.html`). Шаблон:

```xml
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <rule name="API proxy" stopProcessing="true">
          <match url="^api/(.*)" />
          <action type="Rewrite" url="http://localhost:3000/api/{R:1}" />
        </rule>
        <rule name="Static fallback" stopProcessing="true">
          <match url=".*" />
          <conditions logicalGrouping="MatchAll">
            <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true" />
          </conditions>
          <action type="Rewrite" url="/index.html" />
        </rule>
      </rules>
    </rewrite>
  </system.webServer>
</configuration>
```

- **T5.5** HTTPS: получить сертификат (на Windows удобно `win-acme` для Let's Encrypt),
  привязать к сайту в IIS (https-биндинг на 443). Включить редирект http→https.
- **T5.6** App Pool и права: у пула/службы должны быть права на чтение статики; Node-служба
  и IIS на одной машине — прокси идёт на `localhost`, наружу порт 3000 открывать не нужно.
- **T5.7** Доступ Node → MS SQL: если на одной машине — локально. Если на разных — включить
  TCP/IP в SQL Server Configuration Manager, смешанную аутентификацию, открыть порт 1433,
  в строке подключения `encrypt: true` + `trustServerCertificate: true` для самоподписанного.
- **T5.8** Написать `server/DEPLOY-IIS.md`: пошагово всё вышеперечисленное + где служба
  берёт env + настройка регулярных бэкапов MS SQL (теперь это единственная копия данных
  всех пользователей).
- **Готово, когда:** по доменному адресу через HTTPS открывается сайт; Node работает службой;
  `/api` проксируется; есть `web.config` и `DEPLOY-IIS.md`.

## T6 — Тестирование

- **T6.1** Зелёные чистые node-тесты ядра (общие для обеих версий).
- **T6.2** API-тесты на эндпоинты (`npm run test:api`): запрос → ответ → состояние БД.
- **T6.3** Тест параллельного доступа: несколько клиентов одновременно пишут — данные не
  теряются и не путаются.
- **T6.4** Десктоп-смоуки на сборке из `main` зелёные.
- **T6.5** Проверить скрипт переноса данных (T2.7) на копии боевой базы: счётчики сходятся,
  GeoJSON валиден.
- **Готово, когда:** обе версии проходят свои тесты; параллельная работа корректна; перенос
  данных воспроизводим.

## T7 — Релиз

- **T7.1** Staging (отдельный поддомен + тестовая БД), полный прогон сценария на IIS.
- **T7.2** Базовый мониторинг: логи службы, доступность, ошибки эндпоинтов.
- **T7.3** Если делимся ядром: обратный мерж только T1 в `main` — и только после того, как
  `npm run dist:win` из объединённой ветки даёт рабочий `.exe` и смоуки зелёные. Веб-код в
  `main` не вливается.
- **Готово, когда:** веб доступен пользователям на IIS; десктоп продолжает выпускаться из
  `main` без изменений в своём процессе.

## Карта зависимостей

T0 → T1 (фундамент). Затем T2 и T3 можно вести параллельно поверх ядра. T4 встраивается в
backend по ходу. T5–T7 — после готовности T2–T4. Перенос данных (T2.7) нужен до боевого
запуска, но репетируется на копии в любой момент после T2.3.
